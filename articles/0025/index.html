Auditing Java builds with IBM Rational ClearCase and Apache Ant

This article describes how to use the Apache Ant build audit listener that is delivered with ClearCase version 7 and later.

Introduction

Build Auditing refers to the capability whereby during the build process, the versions of all the source files used, the compilation settings and the environment are automatically recorded. With ClearCase you can achieve this only if you are using dynamic views, as the ClearCase Multi-Version File System (MVFS) that is used with dynamic views, audits all the low-level system calls (opens, reads, writes) that are performed on ClearCase data. This capability was primarily designed for C or C++ builds carried out using the clearmake tool; however it can still be used for Java builds. In this article I will describe the build auditing capability in more detail and how it can be used for auditing Ant initiated Java builds. First however, I will explain some of the concepts behind build auditing: derived objects and configuration records.

Derived objects

A derived object is any file that is created by a build script under the control of one of the ClearCase clearmake, omake or clearaudit commands. Derived objects can only be created in dynamic views and they can be shareable or non-shareable. During a clearmake or omake audited build, information about any newly built derived objects is written to the ClearCase VOB. This information can then be used by future builds to determine whether the derived object can be re-used, thus avoiding rebuilding it and potentially decreasing the build time. The process of pulling in a derived object already built in another view is called “winking in” and derived objects created in this way are thereby called shareable.

Key Concepts

MVFS - Multi-Version File System used by ClearCase to audit file system operations.

dynamic view - a "virtual" networked file system that uses MVFS and allows build auditing to be carried out

derived object - object created as a result of a build, usually a binary but can be any "derived" file.

configuration record - data that describes how a derived object was created, i.e. the code versions that went into it, the build script and so on.

As shareable derived objects can only be created and re-used when a build is invoked via the clearmake or omake commands you have to write makefiles to take advantage of this capability. This is obviously not something today’s Java developers are usually prepared to undertake. However, an audited build invoked by the clearaudit command or the ClearCase Ant “build audit listener” can also create derived objects. These derived objects are non-shareable as they cannot be automatically re-used by builds initiated in other views. However the information about the derived object, such as each of the file versions that went into producing it, can still be reported on and used.

The cleartool lsdo command is used to find and query on derived objects. This command displays the fully qualified names of all derived objects created in the current VOB directory regardless of the view in which the derived object was built. As an example, for a traditional C++ project, cleartool lsdo might produce the following output:

>M:\build_view\CPPBank\>cleartool lsdo

14-Sep.19:56   "Account.o@@14-Jan.19:56.105"

In this example, you can see a number of derived objects, either Windows executables or object files. As in this case, an environment may consist of many different derived objects that are essentially variants of the same file, for example multiple instances of an executable that has been built but with different versions of code. To distinguish between them, each derived object has an extended name in the form filename@@unique-identifier, for example:

BankTest.exe@@14-Jan.19:56.108

Each derived object also has an associated configuration record which describes how it was constructed. Note you might see a different variant of identifier (e.g., BankTest.exe@@2006--01-14T19:56.108) depending on what environment you are working in. I will discuss configuration records next.

Configuration records

A configuration record is a compiled object and is not viewable using standard operating system commands. Instead you must use the cleartool catcr <derived object identifier> command, for example:

>cleartool catcr Account.o@@15-Jan.20:28.110

Note that you will only generate a configuration record if you use clearmake, omake or clearaudit to initiate the build. However clearaudit has some auditing limitations compared to the two others (this is indicated in the following list). In general therefore, a configuration record consists of the following sections.

  • Header section
    Shows what, who, where, when, and what view initiated the build.
  • MVFS section
    Shows all the MVFS files and directories read during the build:  versioned objects and view private files (including checked-out versions) as well as all the derived objects produced by the target rebuild.
  • Non-MVFS section
    Shows any declared dependencies outside a VOB - identified by a date and time stamp.
  • Variables and Options section (clearmake/omake only)
    Show any macros defined during the build.
  • Build Script section (clearmake/omake only)
    Shows the actual script that was read from the makefile and executed by clearmake or omake.

There are various options you can use to customise the display of the configuration record, as well as an additional and sometimes very useful command cleartool diffcr to compare two derived objects. Essentially, this command can be used to show you the difference (in ClearCase element versions) between derived objects.

Auditing Ant builds with ClearCase

In order to audit your Ant based Java builds you can use the ClearCase Ant “build audit listener”. This is an Ant library which audits aspects of your build process, allowing you to create derived objects and configuration records for Java builds. Note however, that this capability does not give you automatic build avoidance capabilities. The listener comes bundled with ClearCase as the file CCAudits.jar and is held in the ClearCase bin directory. In order for Ant to execute this listener successfully, you will need to add the path to the CCAudits.jar file to your Java CLASSPATH. For example on Windows, your CLASSPATH should be set similar to the following:

CLASSPATH=%CLASSPATH%;C:\Program Files\Rational\ClearCase\bin\CCAudits.jar

Whereas on Linux or Unix, it should be set similar to the following:

CLASSPATH=$CLASSPATH:"/opt/rational/clearcase/java/lib/CCAudits.jar"

On Windows you can set this configuration permanently by setting the environment variables using the System Applet in the Control Panel; on Linux or Unix you can add the CLASSPATH setting to your login script. Once you have specified a suitable CLASSPATH, you must then decide which Ant tasks you want to audit. You can specify the set of tasks by adding the following line to your default.properties or build.properties file:

CCAudits.auditable_task_list=javac:jar:war

This example will audit the javac, jar and war tasks (if you do not specify any values for this property, the default audited task list is the javac, javah, jar, war and ear tasks). To execute an audited build, you would simply execute your normal Ant target but specify the “build audit listener” as follows:

>ant –listener CCAudits dist

This would execute an audited build of the Ant dist target.

When a build is executed in a dynamic view and using the “build audit listener” every build output (i.e. Java class files) is recorded as a derived object. A number of additional derived objects are also created in the CCAudits directory where the build was executed from. This directory contains a set of derived objects named after your project and audited targets. For each build audit ClearCase creates a single configuration record, recording all the source versions used. All MVFS files read during execution of the audited shell are listed as inputs to the build and all MVFS files created become derived objects, associated with the single configuration record. The only caveat to this process is that the configuration information will only be available as long as the derived objects are in existence. If you subsequently execute an Ant clean, then the derived objects will be removed. However, you can check-in (or stage) the derived object to preserve this information.

Earlier on I mentioned about being able to compare configuration records. Well let us suppose that you used the process described above and now have two derived objects called X and Y. For the second build you knew that a number of elements had been changed but you were not quite sure exactly which ones. To find out this information you could use the cleartool diffcr command as follows:

>cleartool diffcr –flat –ele –s X Y

The arguments to the diffcr command (-flat –ele) produce a nicely formatted output of just the ClearCase elements that have changed. The output from this command would be similar to the following:

<     1 \src\com\ratlbank\model\
       Account.java@@\main\2 <2006-01-08T18:30:11-02>
>     1 \src\com\ratlbank\model\
       Account.java@@\main\4 <2006-01-08T19:36:48-05>
----------------------------
<     1 \src\com\ratlbank\model\
       Customer.java@@\main\2 <2006-01-01T14:36:12-03>
>     1 \src\com\ratlbank\model\
       Customer.java@@\main\3 <2006-01-08T18:06:48-05>

As you can see, the difference between these two builds and that has been highlighted is with the Java files Account.java and Customer.java, which have had new versions implemented in the later build.

In use

Ant build auditing is a feature that is unique to ClearCase. It is undoubtedly a powerful and "interesting" capability, however you will need to assess whether it can be a productive part of your build process. Executing builds in dynamic views will increase the total build time, so if build time is your most important requirement then you would find it hard to justify this capability. However if you were working in a secure or audited environment, then the capability that build auditing provides can go a long way to helping you achieve your aims. From a pragmatic point of view, you might be best executing your release builds using Ant build auditing, whereas day to day integration builds - which tend to be thrown away - can be executed using snapshot views as  normal.
